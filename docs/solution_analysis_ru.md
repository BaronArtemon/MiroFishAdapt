# Аналитика по решению MiroFishAdapt

## 1) Что делает система в целом

Система строит **цифровой графовый мир** по загруженным текстам (PDF/MD/TXT), затем генерирует профили агентов и запускает многoагентную симуляцию в социальных средах (Twitter/Reddit), после чего формирует аналитический отчёт и даёт возможность интерактивного диалога с агентами.

Высокоуровневый pipeline:
1. Загрузка файлов + генерация онтологии сущностей/связей.
2. Построение графа в Zep (GraphRAG память).
3. Извлечение сущностей из графа + генерация профилей OASIS.
4. Генерация конфигурации симуляции через LLM.
5. Запуск симуляции (Twitter/Reddit/parallel) в отдельных Python-процессах.
6. Логирование действий агентов и обновление памяти.
7. Генерация отчёта Report Agent с tool-calling.

## 2) Где и чем выполняются вычисления

### 2.1 Frontend (Vue + Vite)
- UI реализован на Vue 3 + vue-router, сборка Vite.
- Клиент делает REST-вызовы к Flask backend на `/api/graph`, `/api/simulation`, `/api/report`.
- На фронте в основном: визуализация, отображение прогресса, запуск/опрос задач, рендер графа и чатов.

Итог: «тяжёлые» вычисления не на фронте, а в backend + внешних сервисах.

### 2.2 Backend (Flask)
- Flask-приложение (CORS, blueprints, request logging) — оркестратор сценария.
- Асинхронность сделана через фоновые threads для долгих операций (подготовка, построение графа, генерация отчёта).
- Состояние задач/проектов/симуляций сохраняется в файлах (`uploads/...`) и частично в памяти.

### 2.3 Внешние вычислительные контуры
- **LLM API (OpenAI-compatible endpoint)**: онтология, профили, параметры симуляции, отчёты, tool reasoning.
- **Zep Cloud**: графовая память/поиск, хранение графа, чтение узлов/рёбер, поиск контекста.
- **OASIS + camel-ai**: непосредственно модель поведения агентов и симуляция действий по раундам.

Ключевая идея: локальный backend — orchestration + I/O, а интеллект/GraphRAG/соц-симуляция в основном опираются на внешние API и библиотеки.

## 3) Принцип работы по шагам

### Шаг A. Построение знаний (Graph Build)
1. Пользователь загружает документы + формулирует simulation requirement.
2. Backend извлекает текст, генерирует онтологию сущностей/отношений.
3. `GraphBuilderService` создаёт граф в Zep, дробит текст на chunks и отправляет батчами.
4. После обработки Zep возвращаются метрики графа (узлы/связи/типы).

### Шаг B. Подготовка симуляции
1. `SimulationManager` читает сущности из Zep и фильтрует по типам.
2. `OasisProfileGenerator` создаёт профили агентов (может использовать LLM, поддерживается параллелизм по профилям).
3. `SimulationConfigGenerator` через LLM строит параметры симуляции.
4. Файлы профилей и `simulation_config.json` сохраняются в `uploads/simulations/<simulation_id>/`.

### Шаг C. Запуск симуляции
1. `SimulationRunner` поднимает subprocess на один из скриптов (`run_twitter_simulation.py`, `run_reddit_simulation.py`, `run_parallel_simulation.py`).
2. Скрипты OASIS выполняют раунды действий агентов, пишут `actions.jsonl`, `run_state.json`, служебные логи.
3. Runner мониторит прогресс, агрегирует действия по платформам, поддерживает pause/stop/status.
4. Через файловый IPC (commands/responses JSON) возможны interview-команды к агентам во время/после прогона.

### Шаг D. Пост-аналитика и отчёт
1. `ReportAgent` планирует структуру отчёта.
2. Для каждого раздела в ReACT-цикле вызывает инструменты Zep (поиск/панорама/insight/interview).
3. Формирует итоговый markdown/структурированный результат и подробный `agent_log.jsonl`.

## 4) Технологический стек

### Frontend
- Vue 3, Vue Router, Axios, D3, Vite.

### Backend
- Python 3.11+, Flask, Flask-CORS, Pydantic.
- OpenAI SDK (совместимые endpoint’ы), zep-cloud SDK.
- camel-oasis + camel-ai для симуляции.
- PyMuPDF/chardet/charset-normalizer для обработки документов.

### Infrastructure / Runtime
- Локальный запуск через npm + uv, либо Docker Compose.
- Хранилище состояния — файловая структура в `backend/app/uploads`.

## 5) Где узкие места и какие нужны мощности

### 5.1 Что нагружает систему сильнее всего
1. **LLM-вызовы** (генерация профилей, конфигов, отчётов) — главный фактор latency и стоимости.
2. **Размер и сложность графа в Zep** — влияет на время индексации и retrieval.
3. **Количество агентов × число раундов** в OASIS — рост CPU-времени и длительности симуляции.
4. **Interview/Report tool loops** — дополнительные итерации LLM + retrieval.

### 5.2 Распределение нагрузки
- CPU/GPU локально: в основном Python-оркестрация и процессы OASIS (CPU-bound умеренно/высоко при больших мирах).
- Сеть: интенсивные вызовы к LLM API и Zep Cloud.
- Диск: логи (`actions.jsonl`, `agent_log.jsonl`), состояния задач, профили/конфиги.

### 5.3 Практические оценки мощностей
- **PoC/демо (10–50 агентов, 10–40 раундов):**
  - 4 vCPU, 8–16 GB RAM, SSD.
  - Узкое место чаще сеть + квоты LLM/Zep.
- **Рабочий режим (100–300 агентов, 50–200 раундов):**
  - 8–16 vCPU, 16–32 GB RAM, SSD NVMe.
  - Желательны лимиты на concurrency профилей/интервью.
- **Тяжёлый режим (300+ агентов, длинные сценарии):**
  - 16+ vCPU, 32–64 GB RAM, вынесение воркеров/очередей, строгий rate-limit и бюджет LLM.

> Важно: GPU как обязательный компонент не требуется, т.к. LLM-инференс вынесен во внешний API. Если переходить на self-hosted модели, профиль железа изменится радикально.

## 6) Принцип надежности и наблюдаемости

- Долгие операции вынесены в фоновые задачи с прогрессом.
- Состояния задач/симуляций сериализуются в JSON, есть возможность восстановить статус после рестарта.
- Симуляции пишут детальные action-логи по платформам.
- Для отчётного агента сохраняется подробный JSONL лог каждого шага ReACT.

## 7) Ключевые риски и ограничения

1. **Внешняя зависимость от API** (LLM/Zep): квоты, rate-limits, сетевые задержки.
2. **Стоимость**: профили + отчёты + интервью могут генерировать много токенов.
3. **Согласованность состояния** при аварийных остановках subprocess’ов.
4. **Линейный рост времени** при увеличении агентов/раундов без горизонтального масштабирования.

## 8) Рекомендованная модель эксплуатации

1. Ввести бюджетирование токенов и максимум tool-call per report section.
2. Ограничить parallel profile generation по ресурсу инстанса.
3. Перенести фоновые работы в очередь (Celery/RQ) при промышленной нагрузке.
4. Делать batch-run сценариев и кэшировать retrieval-ответы для отчётов.
5. Ввести SLO: время graph build, prepare, round throughput, report latency.
